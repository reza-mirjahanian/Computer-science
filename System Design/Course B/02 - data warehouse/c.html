01 · The Problem

Running analytics on your production database is a recipe for disaster.

Most apps store data in OLTP databases. They're great for fast writes. They're terrible when the CEO asks "what were our top 10 products last quarter?"

Analytics kills the app

A single aggregation query locks tables. Users feel it. The dashboard wins, the customer loses.

Data is scattered

Orders in Postgres. Events in Kafka. CRM in SaaS. Spreadsheets on laptops. No single view.

No history

OLTP stores the current state. The warehouse stores every state. Trend needs time.

Different teams, different views

Finance, marketing, product — each needs curated data. Not raw tables. Not 47 joins.

02 · Architecture

Four stages. One direction.

Sources feed in. The warehouse models and serves. BI tools consume. Every layer has a job.

Live data flow
1
Sources

Heterogeneous systems. Different formats. Different speeds. Each speaks its own dialect.

2
Ingestion

Extract, transform, load. Or extract, load, transform (ELT). Tools: Airflow, dbt, Fivetran.

3
Warehouse

Columnar storage. Massively parallel processing (MPP). Modeled as star or snowflake.

4
Consumers

BI tools, ML pipelines, reports, ad-hoc SQL. All speak the same SQL dialect.

03 · How It Works

A query travels through five layers.

From raw source to insight on a dashboard — here's the journey of one row.

01

Extract

CDC · batch · API

Pull data from sources. Use CDC (change data capture) for low-latency, or batch for simplicity. Don't hammer the source — backoff is your friend.

02

Stage

raw · immutable

Land raw data unchanged in a staging schema. This is your rewind button. If downstream logic breaks, you can replay from here.

03

Transform

dbt · Spark · SQL

Clean, dedupe, cast types, join, aggregate. Build dimension and fact tables. Make data trustworthy — not just available.

04

Serve

marts · semantics

Expose curated tables in data marts — one per business domain. Add a semantic layer so "revenue" means the same thing everywhere.

05

Consume

BI · ML · alerts

Dashboards, notebooks, ML features, scheduled reports. Cache hot queries. Throttle expensive ones. The warehouse is a shared resource — protect it.

04 · Modeling

The star schema is the warehouse's native shape.

One fact table in the middle. Dimensions radiate out. Few joins, fast scans, easy for humans to read.

Star schema · fact_sales

Fact table

Holds events and measures. Tons of rows. Mostly numeric. Foreign keys to dimensions. Grows fast.

Dimension tables

Holds context. The "who, what, where, when." Smaller. Mostly text. Changes slowly.

Why it works

Columnar engines love wide, denormalized dimensions. One hop from fact to dim. The optimizer can prune aggressively.

Example query · top categories last quarter
-- Revenue by category, last quarter
SELECT
  p.category,
  c.region,
  SUM(f.quantity * f.unit_price) AS revenue
FROM fact_sales    f
JOIN dim_product  p USING (product_id)
JOIN dim_customer c USING (customer_id)
JOIN dim_date      d USING (date_id)
WHERE d.quarter = '2024-Q4'
GROUP BY p.category, c.region
ORDER BY revenue DESC
LIMIT 10;
dbt model · dimension with SCD Type 2
-- models/dim_customer.sql
WITH src AS (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY id ORDER BY updated_at
  ) AS rn
  FROM staging.customers
)
SELECT
  id          AS customer_key,
  id          AS customer_id,
  email, segment, region,
  VALID_FROM(updated_at) AS valid_from,
  VALID_TO(updated_at)   AS valid_to
FROM src WHERE rn = 1
05 · Trade-offs

Powerful. Expensive. Not for everything.

A warehouse earns its cost when analytics matter at scale. For tiny data or pure blob storage, look elsewhere.

Strengths

  • →Scan speed: columnar + MPP scans billions of rows in seconds.
  • →Single source of truth: everyone queries the same modeled tables.
  • →Historical depth: years of snapshots, not just current state.
  • →ACID guarantees: reliable, consistent, auditable.
  • →SQL ecosystem: every BI tool speaks SQL. Plug and play.
  • →Governance: row-level security, lineage, access control built in.

Costs & limits

  • →Price: storage + compute is expensive at scale. Idle warehouses still cost.
  • →Latency: batch ETL means data is minutes to hours stale. Not real-time.
  • →Schema rigidity: schema-on-write hurts when sources change often.
  • →ETL complexity: pipelines break. Backfills hurt. Maintenance is real work.
  • →Bad fit for unstructured: images, video, free text — use a lake instead.
  • →Not for OLTP: no high-frequency writes. Don't put your app behind it.

Recommended

Lakehouse pattern

Lake for raw + warehouse for curated. Use Iceberg / Delta Lake to get ACID on object storage. Best of both worlds.

Modern stack

Cloud-native warehouse

Snowflake, BigQuery, Redshift. Decoupled storage & compute. Scale to zero. Pay per query.

Avoid

OLAP on OLTP

Don't run heavy aggregations on your production Postgres. Use a replica at minimum — but a real warehouse is the right answer.

07 · Common Mistakes

Where warehouses go wrong.

Most warehouse failures aren't engine problems. They're modeling, governance, and pipeline problems.

3NF everywhere

Normalizing for analytics means 12 joins per query. The optimizer suffers. Use star schemas instead.

Fix: model as star or snowflake.

No SCD strategy

Customer upgrades from "free" to "pro". Your historical revenue now attributes to "pro". History is broken.

Fix: Slowly Changing Dimensions Type 2.

Loading unvalidated data

Bad source data flows straight into fact tables. Dashboards show negative revenue. Trust dies fast.

Fix: tests in dbt. Reject bad rows before publishing.

No partitioning

Fact table is 5 billion rows. Every query scans all of them. Credit card bill explodes.

Fix: partition by date. Cluster by common filter.

Two definitions of "revenue"

Finance says $4M. Marketing says $5M. Same data, different filters. Meetings turn into forensics.

Fix: centralized semantic layer + metrics.

Treating it as real-time

Stakeholders demand "fresh" data. You shrink the batch window to 1 minute. Pipeline thrashes. Costs explode.

Fix: use a streaming layer (Kafka + materialized views) for hot paths.
08 · Key Takeaways

Remember these seven things.

01

Separate OLTP from OLAP

They optimize for opposite things. Mixing them compromises both.

02

Columnar beats row for scans

Read only the columns you need. Compression is better. Aggregations fly.

03

Star schema is the default

Fact in the middle. Dimensions around. Easy for humans, fast for engines.

04

Partition by time

Most queries filter on date. Partitioning prunes 99% of data instantly.

05

Model for trust, not just access

Tests, docs, lineage. If users don't trust the data, the warehouse is dead weight.

06

Lakehouse is the modern shape

Open table formats (Iceberg, Delta) blur the lake / warehouse line. Plan for both.

07

Cost is a first-class metric

Every query costs money. Monitor, throttle, cache. Treat compute like the scarce resource it is.

08

ELT > ETL in the cloud

Load raw, then transform in the warehouse with SQL. Faster iteration, cheaper compute.

09

One semantic layer to rule them all

Define "revenue" once. Every tool sees the same number. Stop the metric wars.
