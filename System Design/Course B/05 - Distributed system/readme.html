A distributed system is a collection of independent computers that appear to users as a single coherent system. Each computer is a node.

01 · THE PROBLEM

Why not just one big computer?

A single machine has limits: CPU, memory, disk, network. When traffic grows, you either scale up (bigger box) or scale out (more boxes). Distributed systems choose the latter.

📈

Scale

Add nodes to handle more traffic. Horizontal scaling is cheaper and more flexible than buying a bigger server.

🛡️

Reliability

No single point of failure. If one node dies, others keep serving. Users don't notice.

⚡

Latency

Place nodes close to users around the world. A request from Tokyo shouldn't travel to Virginia.

02 · HOW IT WORKS

A request, step by step

Follow one user request as it travels through a distributed system.

1
Client → DNS

User types api.app.com. DNS returns the IP of a load balancer.

2
Load Balancer

Picks a healthy node using round-robin, least-connections, or consistent hashing.

3
Node processes

The chosen node runs business logic, reads cache, queries DB, and builds a response.

4
Replication

Writes are replicated to peer nodes asynchronously for durability and read scaling.

04 · THE FUNDAMENTAL TRADE-OFF

CAP Theorem

In a network partition, a distributed system can only guarantee two of three properties.

🏦
CP — Consistency + Partition

Refuse to answer if data might be stale. Used in banks, payment systems, ZooKeeper, etcd, HBase.

🛒
AP — Availability + Partition

Always answer, even with stale data. Used in shopping carts, social feeds, Cassandra, DynamoDB.

⚖️
CA — Consistency + Availability

Only possible in a single-node system. Not realistic for distributed setups.

05 · TRADE-OFFS

Monolith vs Distributed

Distributed systems are powerful — but they come with real costs. Choose deliberately.

Aspect Monolith Distributed
Deployment One binary, easy Independent services, CI/CD per service
Scaling Vertical only Horizontal, per service
Fault isolation One bug kills all Blast radius contained
Debugging Simple stack traces Needs tracing, logs aggregation
Data consistency ACID transactions Eventual consistency, sagas

06 · COMMON MISTAKES

What breaks in production

Distributed systems fail in subtle ways. Here are the anti-patterns that bite engineers the most.

⚠️
Treating the network as reliable

Networks drop packets, partitions happen, nodes die. Always set timeouts, retries with backoff, and circuit breakers.

// Go: retry with exponential backoff
for i := 0; i < 3; i++ {
  err := call(ctx)
  if err == nil { return nil }
  sleep(backoff(i))
}
⏱️
Ignoring clock skew

Nodes disagree on time. Don't use Date.now() for ordering across nodes. Use Lamport clocks, Hybrid Logical Clocks, or logical IDs.

🔁
Non-idempotent writes

Retries on timeouts can double-charge users. Make every write idempotent with a request ID or use idempotency keys.

// HTTP header
Idempotency-Key: "req_8f3a2b"
// Server returns same response on retry
🧩
Distributed transactions

Two-phase commit blocks the system. Prefer the Saga pattern: a sequence of local transactions with compensating actions.

🔍
No observability

Without traces, metrics, and logs, debugging is a guessing game. Instrument every hop with a correlation ID.

📦
Premature microservices

Start monolithic. Split only when you have clear bounded contexts and real scaling pressure. Complexity is a cost.

07 · KEY TAKEAWAYS

10 lessons to remember

01

A node is any independent machine in the system — server, container, or process.

02

Scale horizontally by adding nodes, not by buying bigger hardware.

03

CAP theorem is real: pick CP or AP based on your business, not on hype.

04

Assume the network will fail. Timeouts, retries, and circuit breakers are mandatory.

05

Use load balancers and health checks to route traffic away from bad nodes.

06

Cache aggressively (Redis), but design for cache invalidation from day one.

07

Prefer eventual consistency where possible; it unlocks massive scale.

08

Make every write idempotent — retries should be safe.

09

Invest in observability: distributed tracing, structured logs, SLOs.

10

Simplicity wins. Start with a monolith, split only when you feel the pain, and measure every decision.
