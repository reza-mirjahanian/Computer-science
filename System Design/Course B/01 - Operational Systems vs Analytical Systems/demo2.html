01 · The Problem

One database. Two masters. Both lose.

Production databases serve live users. Analysts serve the business. Each wants the opposite of the other.

Lock Contention

A heavy GROUP BY scan locks rows. Checkout requests stall. Users see timeouts.

Schema Mismatch

Normalized for writes (3NF). Denormalized for scans (star schema). You cannot optimize for both at once.

Different Time Horizons

Users need now. Analysts need 3 years of history. Retention rules conflict.

04 · The Architecture

How they fit together.

Operational and analytical systems are connected by a pipeline. Changes flow from OLTP to OLAP — never the other way around.

Data flows left to right. The analytical side never writes back to the operational side.
05 · Request Flow

06 · Storage · Why It Matters

Row vs columnar storage.

The single biggest physical difference between OLTP and OLAP. Format follows function.

ROW-BASED · OLTP good for point reads
COLUMNAR · OLAP good for scans
ROW STORAGE
Reads entire rows. Wastes I/O when you only need one column.
COLUMNAR
Reads only the columns you ask for. 10–100x faster scans.
COMPRESSION
Similar values cluster together. Run-length + dictionary encoding shrink data 5–10x.

08 · Why Split Them

Benefits of separation.

Two systems cost more to operate. But they pay for themselves fast.

Performance Isolation

A runaway analytics query cannot lock up checkout. Each side protects the other.

Independent Scaling

Scale OLTP vertically for low latency. Scale OLAP horizontally for scan throughput.

Schema Freedom

Normalize for writes on OLTP. Denormalize into star schema for OLAP. Each gets what it needs.

Cost Optimization

OLTP on provisioned instances. OLAP on serverless, pay-per-scan. Match spend to usage.

09 · Trade-offs

Nothing is free.

Two systems mean two of everything: schemas, pipelines, ops, costs.

PROS
  • Operational stability — analytics cannot break production
  • Right tool for each workload (row vs columnar)
  • Independent scaling and cost tuning
  • Schema evolution decoupled
  • Long retention without bloating OLTP
CONS
  • Data freshness lag (seconds to hours)
  • Pipeline complexity — CDC, ETL, schema drift
  • Two sources of truth — consistency challenges
  • Higher infra cost at small scale
  • Two teams, two mental models, two SLAs
Rule of thumb: If your analytics query takes > 200ms on production, it is time to split. Below that, you may be fine with a read replica.
10 · Common Mistakes

Anti-patterns to avoid.

These will haunt you. Every single one.

ANTI-PATTERN #1

Heavy analytics on the production DB

The CEO runs a 5-year sales report. Checkouts fail. Pager goes off.

→ Fix: Move analytics to OLAP, or at least a read replica.
ANTI-PATTERN #2

Synchronous ETL inside the write path

Writing to OLAP before responding to the user. Latency doubles.

→ Fix: Always async. Use CDC + queue. Never block writes.
ANTI-PATTERN #3

Using OLAP as a source of truth

Reading "current balance" from OLAP for a payment decision. It is stale by minutes.

→ Fix: OLTP is the source of truth. OLAP is a derived view.
ANTI-PATTERN #4

No partitioning on OLAP tables

Billions of rows in one unpartitioned table. Every query full-scans.

→ Fix: Partition by date. Cluster by common filter columns.
ANTI-PATTERN #5

Treating OLAP like OLTP for indexing

Adding B-Tree indexes on every column. Slows loads, does not help scans.

→ Fix: Use sort keys, partition keys, materialized views.
ANTI-PATTERN #6

No schema contract between sides

OLTP changes a column. OLAP pipelines silently break or lose data.

→ Fix: Schema registry. Contracts. Tests on every change.
11 · Modern Approaches ·

The lines are blurring.

Newer systems try to do both. Each has a sweet spot — and a cost.

HTAP

Hybrid TP + AP in one engine

One database, row + columnar storage under the hood. Real-time analytics without a pipeline.

TiDB · SingleStore · CockroachDB · AlloyDB
Best for: real-time analytics on live data
Lakehouse

Open table formats on object storage

Cheap storage, ACID transactions, multiple engines query the same data.

Iceberg · Delta Lake · Hudi · Databricks
Best for: petabyte-scale, multi-engine
Streaming OLAP

Real-time ingest + sub-second queries

Ingest from Kafka directly. Materialized views refresh in milliseconds.

ClickHouse · Materialize · RisingWave · Druid
Best for: dashboards with < 1s freshness
reality: Most companies still run OLTP + OLAP separately, but stream CDC into a lakehouse layer for near-real-time analytics. Best of all worlds — at the cost of complexity.
12 · Practical Example

E-commerce, end to end.

A user orders a coffee mug. Two queries, two systems, one outcome.

OLTP · PostgreSQL ~ 12ms
-- User checks out. Fast, ACID. BEGIN; INSERT INTO orders (id, user_id, total, status) VALUES ('ord_8821', 'u_42', 42.00, 'paid'); INSERT INTO order_items (order_id, sku, qty) VALUES ('ord_8821', 'mug_red', 2); UPDATE inventory SET qty = qty - 2 WHERE sku = 'mug_red'; COMMIT; -- User sees "Order confirmed"
OLAP · BigQuery ~ 2.4s · 1.2 TB scanned
-- Analyst: revenue by category, last 30d SELECT p.category, SUM(oi.qty * oi.price) AS revenue FROM fact_order_items oi JOIN dim_products p USING (sku) JOIN dim_dates d ON oi.order_date = d.date WHERE d.year = 2025 AND d.month = 1 GROUP BY p.category ORDER BY revenue DESC; -- Dashboard refreshes

13 · Key Takeaways

Remember these.

Nine lessons that will save you 3 a.m. pages.

01

Different jobs, different tools

OLTP runs the business. OLAP understands it. Do not make one do both.

02

Row vs columnar

OLTP stores rows for fast writes. OLAP stores columns for fast scans. Format follows function.

03

CDC is the bridge

Stream changes from OLTP to OLAP. Never block writes for analytics.

04

OLTP is the source of truth

Money, inventory, user state — read from OLTP. OLAP is a derived view.

05

Lag is a feature, not a bug

Minutes of staleness on OLAP is usually fine. Decouple for resilience.

06

Never query prod for analytics

At minimum use a read replica. Better: ship to OLAP. Always.

07

Schema is a contract

OLTP and OLAP will diverge. Schema registry + tests prevent silent breakage.

08

Pick the modern path

Lakehouse + streaming OLAP is the default. HTAP for narrow cases.

09

Cost follows the workload

Provisioned for OLTP. Pay-per-scan for OLAP. Match the model or pay 10x.
