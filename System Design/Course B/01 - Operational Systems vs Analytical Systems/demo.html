SYSTEM DESIGN 101

Operational (OLTP) vs Analytical (OLAP) Systems

Two different jobs. Two different databases. Mixing them up is the #1 cause of slow apps and slow dashboards.

1. The Problem

One system runs the business. Another system understands the business. You can't optimize a database for both at the same time.

⚡

Operational (OLTP)

Powers the app right now. "Place this order." "Update this balance." Millions of small, fast read/write transactions.

📊

Analytical (OLAP)

Powers decisions. "What were sales by region last quarter?" Few huge, slow queries scanning billions of rows.

2. Why Engineers Care

Heavy analytics on OLTP locks tables and crashes production checkout.

Wrong storage shape turns a 2-second report into a 2-hour timeout.

Right split = fast checkout AND fast dashboards, simultaneously.

3. How It Works — Architecture

OLTP stores live data row-by-row. A pipeline copies and reshapes it into OLAP, column-by-column, for analytics.

5. Data Flow: Input → Processing → Output

📝

Input

User submits an order → row inserted into OLTP, transaction commits in milliseconds.

🔄

Processing

CDC stream or nightly batch extracts changes, transforms, loads into OLAP warehouse.

📊

Output

Analyst runs a query scanning years of orders → aggregated chart in seconds.

6. Advantages

OLTP strengths

  • ✅ ACID guarantees keep data consistent
  • ✅ Millisecond response for single records
  • ✅ Scales to thousands of concurrent users

OLAP strengths

  • ✅ Scans billions of rows fast (columnar + compression)
  • ✅ Built for complex aggregations & joins
  • ✅ Decoupled from production load

7. Trade-offs

OLTP limits

Bad at scanning huge ranges. Row storage wastes I/O when you only need 2 of 50 columns.

OLAP limits

Bad at single-row updates. Data is usually stale by minutes-to-hours (pipeline lag).

8. Common Mistakes

⚠ Running reports on prod OLTP

A heavy GROUP BY on the orders table locks rows and slows checkout for everyone.

⚠ Treating OLAP as real-time

Expecting a warehouse to reflect data inserted 2 seconds ago. It's batch/streaming, not instant.

⚠ No CDC, only nightly dumps

Dashboards 24h stale during incidents. Use change-data-capture for near-real-time needs.

Practical Example

An e-commerce platform: orders write to Postgres (OLTP). A CDC tool (e.g. Debezium) streams changes into Kafka, which loads into ClickHouse (OLAP) for the revenue dashboard.

-- OLTP: fast single insert
INSERT INTO orders (id, user_id, total, created_at)
VALUES (5821, 901, 79.99, now());

-- OLAP: heavy aggregation, runs in warehouse, not prod
SELECT region, date_trunc('month', created_at) AS month,
       SUM(total) AS revenue
FROM orders_fact
GROUP BY region, month
ORDER BY month;

Key Takeaways

🔹 OLTP = operations, OLAP = analytics. Different jobs.

🔹 Row storage for transactions, column storage for aggregation.

🔹 Never run heavy analytics directly on production OLTP.

🔹 Pipelines (ETL/CDC) connect the two systems.

🔹 OLAP data is eventually consistent, not real-time.

🔹 Star schemas make OLAP queries simple and fast.

🔹 Pick tools by access pattern, not by hype.

🔹 Most real systems need both — together, not instead.
