Architecture Fundamentals

Systems of Record
& Derived Data

Every data system falls into one of two categories. Understanding this distinction is the key to building scalable, reliable architectures.

System of Record (SOR)

The authoritative source of truth. Every piece of data originates here. No other system can override it.

  • ▸ Optimized for writes & integrity
  • ▸ ACID transactions, strict consistency
  • ▸ Examples: PostgreSQL, MySQL, CRM, ERP

Derived Data System

Consumes, transforms, and re-materializes data from the SOR for specific use cases.

  • ▸ Optimized for reads & queries
  • ▸ Eventual consistency acceptable
  • ▸ Examples: Elasticsearch, Redis cache, Data Warehouse

🎯 Why This Matters

Problem: Mixing SOR and derived logic in one system creates a monolith that's hard to scale.
Solution: Separate concerns. Let SORs handle writes; let derived systems handle specialized reads.
Result: Independent scaling, better resilience, and clear data contracts.

How Data Flows

One source of truth → many specialized views

How It Works

Five steps from write to specialized read

1

Write to the System of Record

All mutations go through the SOR. It validates, enforces constraints, and ensures ACID compliance. No other system accepts writes directly.

2

Capture the Change

CDC tools like Debezium tail the database write-ahead log (WAL). Every insert, update, and delete becomes an event — immutable and ordered.

3

Publish to the Event Bus

Events land on a durable log (Kafka). Multiple consumers can read the same event stream independently — each at its own pace.

4

Derived Systems Consume & Transform

Each derived system processes events for its purpose: indexing for search, aggregating for analytics, or caching hot data for low-latency reads.

5

Serve Specialized Queries

Applications query the derived system optimized for their use case. The SOR stays lean, handling only transactional writes.

Trade-offs

Every architectural decision has a cost

Approach Consistency Latency Complexity Best For
CDC (Change Data Capture) Eventual ✓ ms–seconds Medium ✅ Recommended
Dual Writes ⚠️ Risky Instant Low Simple apps (avoid at scale)
Materialized Views Strong ✓ Instant Low Same-database aggregations
Batch ETL Eventual Hours Medium Analytics, reporting
Event Sourcing Eventual ✓ ms–seconds High Audit trails, complex domains

✅ CDC is the sweet spot for most systems: low latency, manageable complexity, and battle-tested tooling.

Common Mistakes

Anti-patterns that cause data corruption and outages

⚠️

Writing to Derived Systems

Never accept writes directly into a cache, search index, or replica. It creates split-brain and data drift.

// ❌ Anti-pattern
redis.set("user:42", data);
// ✅ Correct: write to SOR, let CDC propagate
⚠️

Treating Derived as Source

A search index or cache can go stale or get wiped. Always design so you can rebuild derived data from the SOR.

// Rebuild from SOR anytime
SELECT * FROM orders
WHERE updated_at > :last_sync;
⚠️

Ignoring Replication Lag

Derived systems lag. If you write then immediately read from a derived view, you'll see stale data. Handle this explicitly.

// Read-your-writes: go to SOR
if (needsFreshData) {
  querySOR(); // not cache
}
⚠️

Schema Evolution Blindness

Changing the SOR schema without updating derived consumers breaks pipelines. Plan for backward compatibility.

⚠️

No Rebuild Strategy

If a derived system corrupts, can you rebuild it from scratch? You must have a tested bootstrap process.

⚠️

Over-Deriving

Not every query needs a derived system. Start simple. Add a cache or index only when you measure a real bottleneck.

In the Wild

How industry leaders apply this pattern

🛒

E-Commerce

SOR: Orders DB (PostgreSQL). Derived: Product search (Elasticsearch), inventory cache (Redis), sales analytics (BigQuery).

🎬

Streaming (Netflix)

SOR: Content catalog DB. Derived: Search (Elasticsearch), recommendations model, user history cache.

🏦

Banking

SOR: Core ledger (mainframe/SQL). Derived: Statement PDFs, fraud detection pipeline, mobile app balance cache.

Key Takeaways

Remember these principles

1️⃣

One Source of Truth

The SOR is always right. Everything else is a copy.

2️⃣

Separate Reads & Writes

SOR for writes; derived systems for reads. Scale independently.

3️⃣

Embrace Eventual Consistency

Derived data lags. Design for it. Don't fight it.

4️⃣

Use CDC, Not Dual Writes

Let the database log be your event source. Avoid manual sync.

5️⃣

Always Be Rebuildable

Derived systems are disposable. The SOR is sacred.

6️⃣

Start Simple

Add derived systems only when you hit a real scaling bottleneck.
